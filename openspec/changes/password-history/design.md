## Context

mmpassword 当前在用户修改密码时直接覆盖旧值，无任何历史保留机制。当用户在密码管理器中生成了新密码但尚未在实际网站完成变更时，旧密码永久丢失。

当前数据模型：`Field` 结构包含 `name`, `value`, `field_type`, `protected` 四个字段，无历史记录能力。vault 文件使用 ChaCha20-Poly1305 加密存储，字段变更通过 `update_entry` 命令一次性替换所有 fields。

## Goals / Non-Goals

**Goals:**
- 密码字段值变更时自动保留旧值，最多 3 条历史记录
- 历史记录包含密码值和变更时间戳
- 前端在条目详情中以折叠区域展示历史密码，支持查看和复制
- 向后兼容：旧 vault 文件无需迁移即可正常使用

**Non-Goals:**
- 不提供密码轮转引导式工作流（仅保留历史，不引导用户去网站改密码）
- 不提供定时提醒功能（如"密码已修改 N 天未确认"）
- 不支持非密码类型字段的历史记录
- 不提供手动清除历史密码的功能（由 FIFO 自动淘汰）

## Decisions

### 1. 历史记录存储位置：Field 级别

**选择**：在 `Field` 结构上新增 `password_history: Vec<PasswordHistoryEntry>`

**替代方案**：
- Entry 级别单独字段：无法处理多条目密码字段（虽然罕见），且与 fields 范围不一致
- 独立的历史表/集合：增加数据管理复杂度，需要额外关联逻辑

**理由**：历史记录与具体字段绑定，Field 级别最自然。使用 `#[serde(default)]` 确保向后兼容。

### 2. 历史记录数量：最多 3 条

**选择**：FIFO 队列，上限 3 条

**理由**：3 条足以覆盖"改了但忘了去网站改"的场景，同时不会显著增加 vault 文件体积。

### 3. 字段匹配策略：按 name 匹配

**选择**：更新时通过字段 `name` 匹配新旧字段，而非索引

**理由**：用户可以在编辑时修改字段名称或增删字段，按名称匹配比按索引更可靠。如果找不到同名旧字段，则不产生历史记录。

### 4. 前端展示：InlineField 组件内渲染

**选择**：在密码字段的 InlineField 组件中，viewing 模式下渲染历史密码区域

**理由**：历史密码属于特定字段，视觉上与其所属字段关联最直观。

## Risks / Trade-offs

- **vault 文件体积增加** → 每条密码历史约增加 50-100 字节（未加密前），最多 3 条 × 密码字段数，影响可忽略
- **字段重命名导致匹配失败** → 如果用户同时修改了字段名和密码值，按 name 匹配会失败，不会产生历史记录。这是可接受的边界情况
- **旧版本客户端读取新 vault** → `#[serde(default)]` 确保旧版本忽略 `password_history` 字段，不会报错，但历史数据在旧版本中不可见
